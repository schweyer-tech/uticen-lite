# Workpaper App-Parity Presentation Spec (SDK HTML renderer)

**Date:** 2026-06-17
**Component:** `uticen_lite/render/html.py`
**Status:** Design — canonical spec for the SDK's self-contained HTML workpaper.

## Goal & parity level

Bring the SDK's `render_html(wp)` output **structurally in line with the Uticen app's
workpaper view** (same section names, same order, same metadata blocks, same jump-nav) and
**visually close** via the app's shared design tokens — **not a pixel-exact mirror of the React
app**. The SDK emits a single self-contained HTML file; the app renders an interactive React tree
backed by a richer data model. Parity is defined as: *a reviewer who knows the app recognises the
SDK document as the same artifact.*

### Non-negotiable renderer guarantees (preserve all)

- Single self-contained document: opens `<!doctype html>`; **inline `<style>` only** — no
  `<link rel="stylesheet">`, no external assets.
- **No `<script>` anywhere.** Collapse and jump-nav are achieved with `<details>/<summary>` and
  anchor links only. Scroll-spy active-section highlighting is **omitted** (it would need JS).
- **All author/data-derived text** passes through `html.escape` (existing `_e` helper).
- **Pure stdlib / Pyodide-safe**: no template engine, no pandas/pydantic, no system calls.

## 1. Canonical section model + order

The SDK adopts the app's section **names and order**. The app has 9 sections; several are
app-only (interactive workflow, no SDK data). Decision per app-only section is stated inline.

| # | Anchor id | App section name | SDK source field(s) | Render decision |
|---|-----------|------------------|---------------------|-----------------|
| ★ | `#results` | **Results** | `RunRecord` aggregates across procedures: `population_size`, `passed`, `failed`, `pass_rate`, `violations` count | **Render.** 4 metric tiles (Passed / Failed / Records tested / Exceptions) + verdict pill. Read-only — no editable `results_summary` narrative (app-only); omit that line. |
| 0 | `#objective-scope` | **Objective & scope** | `Workpaper.objective` | **Render objective.** Scope table fields (`nature_of_test`, `period`, `coverage_basis`, materiality, `scope_exclusions`) are app/plan-owned — **omit** them rather than emit empty rows. Single "Full population — no sampling" coverage line stands in for coverage basis (see §5). |
| 1 | `#control` | **Control** | `Workpaper.control_id`, `Workpaper.title`, `Workpaper.narrative` | **Render** a KV table (Control ID, Title) + the control-level `narrative` as the design/reliance text. Owner / risk ratings / assertion chips are app-only metadata — **omit** (not in SDK model). |
| 2 | `#data-sources` | **Data sources** | `RunRecord.provenance: list[SourceProvenance]` (`path`, `sha256`, `row_count`, `source_id`) deduped across procedures | **Render**, collapsible (§3). Each source = one `<details>`: summary header (path + row-count badge), provenance chips (sha256, row count). Completeness tie-out / classification / steward / relevance narrative are app-only — **omit**. |
| 3 | `#procedures` | **Procedures** | `Workpaper.procedures: list[Procedure]` → per proc: `title`, `narrative`, `test_code`, `result` (`RunRecord`) | **Render** one block per procedure: `P{i}` header + PASS/FAIL badge, narrative, collapsible code (§3), results metric line, full-population statement (§5), violations table if any. |
| 4 | `#evaluation` | **Evaluation** | derived counts only: failed-procedure count, total violations | **Render a minimal static read-only summary** (failed procedures, total violations/exceptions). Root-cause narrative + severity-classification selector are app-only interactive — **omit** the editable/selector parts. |
| 5 | `#exceptions` | **Exceptions** | `RunRecord.violations: list[Violation]` (`item_key`, `severity`, `description`, `details`) | **Render**: summary table (E-ref, item key, severity badge) + per-violation collapsible disposition panel (§3) showing `description` + `details`. Disposition form fields (status, remediation owner/due, retest) are app-only — **omit** the form; show data only. |
| 6 | `#conclusion` | **Conclusion** | derived verdict from violations (see §5) | **Render** the verdict line ("Operated effectively" / "Operated with deficiencies") computed from total violations. Conclusion-type selector, residual-risk-revised, reliance narrative are app-only — **omit** the editable parts; emit the computed verdict statement only. |
| 7 | `#sign-off` | **Review & sign-off** | none | **Omit entirely** (cleaner than a placeholder). Sign-off is a live app workflow (signatures, independence check, finalize/lock) with **zero SDK data** — a static placeholder would be misleading in an exported artifact. Instead the document footer carries a single line: `Generated by uticen-lite · {generated_at} · not a finalized workpaper`. The sidebar therefore lists **8** entries (★,0–6), not 9. |

**Summary of app-only handling:** Sign-off → **omit** (footer disclaimer instead). All other
interactive controls (severity selector, conclusion selector, disposition form, editable
narratives, scope/materiality grid) → **render the data-driven/computed portion, omit the
interactive/empty portion.** No empty form fields are emitted.

### Aggregate (Results) computation

`Results` aggregates over all procedures' `RunRecord`s:

- `records_tested = sum(p.result.population_size)`
- `total_failed   = sum(p.result.failed)`  (== total violations)
- `total_passed   = records_tested - total_failed`
- `exceptions     = sum(len(p.result.violations))`  (== `total_failed`; shown as its own tile)
- verdict: see §5.

## 2. Metadata blocks & placement

| Block | Location | Source | Notes |
|-------|----------|--------|-------|
| **Document header** | top, above sidebar+content row | `title`, `control_id`, `generated_at` | `<h1>` title; `control_id` as mono chip; `generated_at` as meta line. |
| **Sticky results bar** | sticky `top:0` strip directly under header, spanning the content column | aggregate (§1) | `RESULT` label + mono metrics `N pass · N fail · N exc` (color-coded) + verdict pill right-aligned. CSS `position: sticky; top: 0`. |
| **Section metadata tag** | inline, right of each `<h2>` heading | static per section | Mono gray tag naming the backing model, e.g. Results `read-only · run_record`, Data sources `provenance`, Procedures `procedures · run_record`, Exceptions `violations`. Cosmetic parity with the app's source tags. |
| **Framework references** | rendered as a sub-block of the **Control** section (id `#control`), not a standalone top-level section | `framework_refs.nist` (flat ref-tag list) + `framework_refs.extra` (grouped `framework: ref` tags) | Keeps the app's 8-section nav clean; framework refs are control metadata. Render `None` when empty. |
| **Verdict / status** | (a) verdict pill in sticky bar; (b) conclusion statement in `#conclusion` | derived (§5) | Same verdict string in both places. |
| **Per-source provenance chips** | inside each Data-source `<details>` body | `SourceProvenance` | `sha256 <first-8>…` chip + `<row_count> rows` badge + full `path` mono line. |
| **Per-procedure integrity line** | under each procedure header | `run.run_id`, `run.executed_at` | Mono line: `run {run_id[:8]}… · {executed_at}`. Green styling. |
| **Full-population statement** | bottom of every procedure's results block | `population_size` | Bold: `Full population tested: {N} record(s). No sampling was applied.` (existing text, preserved verbatim). |
| **Footer disclaimer** | end of `<body>` | `generated_at` | `Generated by uticen-lite · {generated_at} · not a finalized workpaper`. |

## 3. Collapsible behavior (`<details>/<summary>`, script-free)

| Section | Element | Default state | Summary line |
|---------|---------|---------------|--------------|
| **Data sources** | one `<details>` per source | **closed** (`<details>` with no `open`) | `▸ {path}` + `{row_count} rows` badge |
| **Procedures → code** | one `<details>` wrapping each `<pre>` test-code block | **closed** | `▸ code that ran` + `run {run_id[:8]}…` if present |
| **Exceptions** | one `<details>` per violation | **closed** | `▸ E-{i} · {item_key}` + severity badge |
| Results, Objective & scope, Control, Procedures (block bodies/narratives/results), Evaluation, Conclusion | plain `<section>` | **always expanded** | — |

- The `▸` marker is a static char in the `<summary>` text (decorative, `aria-hidden`); native
  `<details>` provides its own disclosure triangle. Do **not** attempt CSS triangle rotation that
  depends on JS — the native marker is sufficient.
- `<summary>` styled `cursor: pointer; list-style: none;` with hover background.
- No `max-height` transition (would be cosmetic only and native `<details>` toggles instantly);
  acceptable for parity.

### Jump-nav sidebar (anchor links, script-free)

- Layout: a top-level flex row — **left**: `<nav class="wp-sidebar">`, **right**:
  `<main class="wp-content">`.
- Sidebar is `position: sticky; top: 0; align-self: flex-start;` width `192px`, `flex: 0 0 192px`,
  `bg #12141f`, right border `1px #1e2235`, padding `12px`.
- Title block at top: workpaper title (`text-primary`, semibold, 13px) + `control {control_id}`
  (mono, 12px, secondary).
- 8 links, one per section, `href="#<anchor-id>"`, in the order of §1.
  - Each link: `<a>` with an **index badge** (`★` for Results; `0`–`6` for the rest — i.e. the
    app's 0-indexed scheme) + label + optional **count badge** (right-aligned).
  - Count badge appears for Data sources (#sources > 0), Procedures (#procedures > 0, red if any
    failures), Exceptions (#violations > 0, red if > 0). Red = `status-critical-muted` bg +
    `status-critical` text.
- **No active/scroll-spy state** (omitted — needs JS). `:target { }` may optionally tint the
  jumped-to `<section>` header, but the sidebar links carry no persistent active class.
- Each top-level section is `<section id="<anchor-id>">` so the anchors resolve.

## 4. Shared design tokens (bake into inline `<style>`)

Replace the current ad-hoc Tailwind-ish hexes with the **app's STYLE.md tokens**. Define as CSS
custom properties on `:root` and reference them, so the palette is auditable in one place.

```css
:root {
  color-scheme: dark;
  /* backgrounds */
  --bg-base:#0b0d17; --bg-surface-1:#12141f; --bg-surface-2:#1a1d2b;
  --bg-surface-3:#222638; --bg-input:#0f1119;
  /* text */
  --text-primary:#e8e9ed; --text-secondary:#8b8fa3; --text-tertiary:#5c6078;
  /* accent */
  --accent-primary:#3b82f6; --accent-muted:rgba(59,130,246,0.125);
  /* status */
  --status-success:#10b981; --status-success-muted:rgba(16,185,129,0.15);
  --status-warning:#f59e0b; --status-warning-muted:rgba(245,158,11,0.15);
  --status-critical:#ef4444; --status-critical-muted:rgba(239,68,68,0.15);
  --status-info:#6366f1; --status-info-muted:rgba(99,102,241,0.15);
  /* borders + radius */
  --border-default:#1e2235; --radius-card:8px; --radius-input:6px; --radius-badge:9999px;
}
```

- **Font stack** —
  sans: `Inter, -apple-system, BlinkMacSystemFont, "Segoe UI", sans-serif`;
  mono: `"JetBrains Mono", "Fira Code", "SF Mono", monospace`.
  Do **not** add an `@import` of Google Fonts (keeps the file fully self-contained with no network
  fetch); rely on the fallback stack. Inter/JetBrains render if locally present, else fall back.
- **Line-heights** (match app type scale): body `22px`/14px, section heading `26px`/18px, card
  title `22px`/15px, caption `16px`/12px, metric `40px`/32px.
- **Table density:** `font-size: 12px` (caption), `th/td padding: 6px 12px` (`py-1.5 px-3`),
  `border-bottom: 1px var(--border-default)`, header bg `var(--bg-surface-2)`, alt-row optional.
  Mono cells (`item_key`, `sha256`, `path`) use the mono stack at `11px`.
- **Code styling (`pre`):** bg `var(--bg-base)`, border `1px var(--border-default)`,
  `border-radius: var(--radius-input)`, `padding: 12px`, mono `11px`, text `#a5b4fc`
  (app's code-listing indigo), `overflow-x:auto`, `white-space:pre-wrap; word-break:break-all`.
- **Severity classes:** critical/high → `--status-critical`; medium → `--status-warning`;
  low → `--status-success`. (Map the SDK `Severity` enum: `critical`,`high` red; `medium` amber;
  `low` green.)
- **Body:** bg `var(--bg-base)`, color `var(--text-primary)`, base padding `24px`.

## 5. Passing control (0 violations) rendering

When `total_failed == 0` across all procedures:

- **Results:** Failed tile = `0`, Exceptions tile = `0`, `pass_rate` shows `100.0%`. Sticky bar
  metrics read `N pass · 0 fail · 0 exc`. Verdict pill = **`Operated effectively`**
  (`status-success-muted` bg, `status-success` text).
- **Procedures:** each procedure header shows the **PASS** badge (`status-success`); no violations
  table is emitted (existing `if run.violations:` guard already suppresses it).
- **Exceptions section:** still rendered, but the summary table body is replaced with a single
  empty-state line: `No exceptions — control operated without deviations.` (`text-secondary`). No
  `<details>` panels.
- **Evaluation:** `Failed procedures: 0 · Open exceptions: 0`.
- **Conclusion:** statement = `Operated effectively. Full population tested with no exceptions.`
- **Verdict derivation (single source of truth):**
  `verdict = "Operated effectively" if total_failed == 0 else "Operated with deficiencies"`.
  Used identically by the sticky pill and the conclusion line. (SDK has no third
  "not effective" state — that is an app-only conclusion type.)

## 6. SDK-only structural regression test

Add to `tests/render/test_render.py` a `TestRenderHtmlAppParity` class (and reuse the existing
`workpaper` fixture, which has 2 violations = a **failing** control). Add a second fixture
`passing_workpaper` (same shape, `violations=[]`). **No live-app fetch, no network, no coupling to
the app deployment** — every assertion runs against the SDK's own returned HTML string.

Assertions:

1. **Sections present & in order.** Extract every `id="..."` of `<section ...>` (regex over the
   rendered string) and assert the list equals exactly
   `["results","objective-scope","control","data-sources","procedures","evaluation","exceptions","conclusion"]`
   in that order. (Asserts the 8-section model and that sign-off is omitted.)
2. **Sidebar anchors.** For each expected anchor id, assert `href="#<id>"` appears in the rendered
   HTML, and that all 8 hrefs appear **before** the first `<section id=` (i.e. the nav precedes
   content).
3. **`<details>` collapse markers.** Assert `<details>` count == `#sources + #procedures +
   #violations` for the failing fixture; assert none carry the `open` attribute
   (`"<details>"` present, `"<details open"` absent → all default-closed).
4. **Key tokens baked in.** Assert the inline style contains the canonical hexes
   `#0b0d17`, `#12141f`, `#3b82f6`, `#10b981`, `#ef4444` and `--accent-primary`; assert the mono
   stack substring `"JetBrains Mono"` and sans `"Inter"` are present.
5. **Passing vs failing render.**
   - Failing fixture: HTML contains `Operated with deficiencies`, a `>FAIL` badge, the violations
     `INV-001`/`INV-002`, and sticky metric `2 fail`.
   - Passing fixture: HTML contains `Operated effectively`, `100.0%`, the empty-state
     `No exceptions`, `0 fail`, and **no** `<table` inside `#exceptions` (no violation rows).
6. **Guarantees still hold (regression guard).** Re-assert `startswith("<!doctype html>")`,
   `"<script" not in html.lower()`, `'rel="stylesheet"' not in html`, `"<link" not in html`,
   and the existing XSS escape (`"<b>" not in html` / `"&lt;b&gt;" in html`).
7. **Full-population statement per procedure.** `"No sampling was applied"` count ==
   `len(wp.procedures)`; `"100 record(s)"` present.

These are pure string assertions on `render_html(wp)` — deterministic, offline, Pyodide-safe.

## 7. README / docs parity paragraph

> The SDK's `render_html()` produces a workpaper that is **structurally equivalent and visually
> close** to Uticen's in-app workpaper view: the same section model and order (Results,
> Objective & scope, Control, Data sources, Procedures, Evaluation, Exceptions, Conclusion), the
> same sticky results bar and jump-nav sidebar, and the same shared design tokens (dark enterprise
> palette, Inter/JetBrains Mono). It is **not** a pixel-exact mirror of the React app, and it is
> intentionally static: collapsible sections and the sidebar work without any JavaScript
> (`<details>/<summary>` + anchor links), the whole document is a single self-contained HTML file
> with inline styles and no scripts, and interactive/finalization features that exist only in the
> app (sign-off, severity selectors, disposition forms) are omitted because the SDK has no data for
> them.
