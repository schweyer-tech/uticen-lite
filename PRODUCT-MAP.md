# Product Map — uticen-lite

> Present-state inventory of what the SDK **ships now**. Planned/unbuilt work lives in GitHub issues,
> not here. Update a row when a surface ships or changes; remove a row when a surface is retired.
> Last updated: 2026-06-28.

| Surface | Type | What it does |
| --- | --- | --- |
| Distribution / install | docs + release CI | Corporate-ready install paths — pipx, internal index (`--index-url`/`--extra-index-url`), and pinned-wheel/air-gapped — documented in `docs/INSTALL.md`. Wheel is pure-Python `py3-none-any`, ships the web assets + bundled Northwind demo (verified by `tests/plane/test_wheel_build.py`). A manual, token-gated `release.yml` (`workflow_dispatch`, `PYPI_API_TOKEN`) builds + `twine check`s the dist; **not yet published to PyPI**. |
| `controlplane` | command / web app | Launches the local SQLite-backed FastAPI + HTMX control plane (localhost only, default `:8765`); the primary authoring surface. `pip install 'uticen-lite[plane]'` → `controlplane --project <dir>` (or the equivalent `python -m uticen_lite.plane`). |
| Control plane — First-run setup | view | Shown when an engagement has no name yet: name the engagement + pick a framework, or one-click **Load the Northwind demo** (bundled in the wheel, copied into the engagement so it runs immediately). |
| Control plane — Dashboard | view | Lists controls + last-run status; entry point to author, run, and export. Engagement name renders in the header. |
| Control plane — Source manager | view | List engagement sources (friendly title + code id); **Add source** offers two on-ramps — **upload** a CSV / Excel (`.xlsx`, with sheet selection) / Parquet file, or **fetch from a URL** (a one-time GET that snapshots a JSON/CSV response to a local file; credentials, if supplied, persist in `controlplane.db` with a plaintext-at-rest warning). Columns are inferred either way. Persisted to `controlplane.db`. |
| Control plane — Source editor (Definition / Data / History tabs) | view | Three server-rendered tabs: **Definition** (title, description, key columns, column mapping); **Data** (current-file record count, read-only paged row preview, a **Mapping check** callout flagging columns that coerced to all-empty for a non-text declared type, per-file **as-of date**, **Refresh data** → review column diff → confirm, archiving the prior file, and for URL-origin sources a **Re-fetch from URL** action that re-runs the snapshot GET and routes through the same diff-review → confirm flow as a file refresh); **History** (every uploaded file version with as-of date, record count, upload time, current badge). Per-file data lineage in a store-only `source_files` table; `sources.extract_date` mirrors the current file's as-of (bundle path unchanged). |
| Control plane — Control editor (Definition / Logic / History tabs) | view | **Definition** tab is metadata-only: title (edited in place via a header pencil), objective, narrative, framework refs, failure thresholds (with an optional **threshold rationale** captured for the audit file — store-only, never in the bundle), and source binding via a searchable single-select picker — a control's **test logic** is authored in the **Logic** tab (next row), never here. **History** lists prior runs (see Run history). |
| Control plane — Logic (Builder / Flowchart / Python tabs) | view | One place to author a control's test logic (`/controls/{id}/logic/...`) — three server-rendered sub-tabs over a single node graph. **Builder**: every control is a node graph (Import · Filter · Join exists/not-exists/inner/left · Test · Custom Python) — a simple rule is `Import→Test`; rule_spec/empty controls open as a derived/scaffold graph (editable, incl. cross-source conditions); raw-Python controls show a notice. **Live row-counts** per node (over the **full population**, kept fast by a content-addressed step cache that recomputes only from an edited step onward) — each count is **clickable → opens that step's actual rows as a full page in a new browser tab** (`target="_blank"`, server-paginated, with a back-to-builder link and a per-step **"Download this step (.xlsx)"** button), plus a pipeline-level **"Export step workbook (.xlsx)"** (one sheet per step + Summary/About). The inspector + exports are localhost-only evidence and never touch the bundle. A **Draft with AI** affordance. **Procedure sections**: the Builder groups cards into a shared **"Inputs & shared steps"** band on top (data sources + any step feeding ≥2 procedures) followed by one **collapsible `<details>` section per procedure**, whose header **is** the procedure editor — a short **code** (P1, P2, …), a human **name**, a free-text **assertion** (e.g. "Segregation of Duties"), a free-text **narrative**, and a **per-procedure failure threshold**. The procedure header owns ALL procedure-level identity; the **Test node card is pure step mechanics** (input, conditions, severity, description, item-key, a "Belongs to ▾" selector, and its own per-step narrative — no procedure title/threshold). A control with a single procedure keeps an **empty code** (the workpaper heading stays the legacy `P1: title` form). Inserting a **Test** inside a section assigns it to that procedure; a **"Belongs to ▾"** select moves a Test between sections; **＋ Add procedure** appends an empty section. Support nodes (Import · Filter · Join) are **derived** — they sit in the shared Inputs band with color chips showing which procedures they feed. Section collapse state persists per control in `localStorage`. Per-procedure **colors** carry through the Builder, the Flowchart, and the rendered workpaper. A pipeline may **fork into ≥2 terminal Test nodes** — one control, N **procedures** sharing an upstream trunk; a procedure may roll up **several Test checks** into one result (violations merge by item-key, annotated with which checks flagged each; population = **distinct items examined** across those checks). The graph (nodes + a `procedures` array) is store-only and **compiles** to a `rule_spec` (pure single-source) or a `test(pop, sources)` string at run/build (a multi-terminal graph compiles to a **union `test()`** for the single `control.test_code`) — bundle shape unchanged. **Flowchart**: a read-only **multi-lane SVG** grouped into **procedure swimlane bands** — a shared Inputs band on top feeding one labeled, colored band per procedure (parallel branches converge at joins; every terminal Test is marked; a per-procedure legend). Each procedure band is **collapsible**: toggling its header server-re-renders the SVG (`?collapsed=<ids>`, HTMX-swapped) with that band reduced to a single **summary box**. **Python**: the read-only **generated-Python** glass-box + a one-way **Convert to Python test**, or the editable **Python escape hatch** (CodeMirror) for hand-written controls. Custom Python nodes are file/source-starved (allowlist AST lint + lexical starvation + hard export gate). |
| Control plane — AI-assisted draft | view | **Draft with AI** in the Logic ▸ Builder + a `/settings/ai` provider/model picker. Provider-agnostic (Anthropic / OpenAI / local Ollama) behind the optional `[ai]` extra, lazy-imported; keys from env vars only; **offline by default**. Drafts a `rule_spec` from the objective + bound-source schema + a data sample and **applies it into the Builder's Test node** (editable in place; persisted only on Save), re-validated (`parse_rule_spec` + run-on-sample) before it can be accepted. |
| Control plane — Run view | view | Per-run results (totals + violations table) with the **rendered workpaper embedded**. |
| Control plane — Run history | view | Per-control run history (`/controls/{id}/history`, linked from the dashboard + control editor): every run with pass-rate/exception counts, each row linking to its run view, plus an inline-SVG **trend chart** (pass-rate line + per-run exception bars with labeled axes). Read-only over stored runs. |
| Control plane — Export | view | Produces the import bundle (zip) for the Uticen app; validates against the contract. |
| Control plane — Settings | view | A **Settings** hub (`/settings`) linking engagement settings — **rename** the current engagement (single-engagement by design) — and the AI provider picker (`/settings/ai`, which names the exact env var per provider). A **Danger zone** offers a confirm-guarded **Reset to demo data** (`POST /settings/reset-demo` → `reset_to_demo`): wipes the engagement, clears `data/` + `target/`, reloads the Northwind demo, and preserves the AI config + update-check toggle — one-click recovery from a corrupted/wedged store. |
| Control plane — Updates / upgrade | view + action | **Settings ▸ Updates** shows the installed version + detected install method and an **opt-in** "check for updates on launch" toggle (default OFF — preserves zero egress) plus a manual **Check now**. When a newer version exists, a dashboard banner offers **one-click self-upgrade**: the app detects the install method (git checkout → `git pull` + reinstall · pipx → `pipx upgrade` · pip → `pip install -U`), spawns a detached helper, and shuts down for a manual re-run. The same routine is the `uticen-lite upgrade [--check] [--yes]` CLI. No bundle impact. |
| `uticen-lite import <yaml> [--into <dir>]` | CLI | One-time import of a YAML project into `controlplane.db`. |
| `uticen-lite run [dir] [--control] [--at]` | CLI | Run controls over the store, full-population; write workpapers + evidence; persist runs. |
| `uticen-lite build [dir] [--out] [--at]` | CLI | Assemble + write the import bundle from the store. |
| `uticen-lite validate [dir]` | CLI | Light DB integrity check (deprecated stub; prefer the web app). |
| `def test(pop[, sources])` | authoring API | Python full-population test; optional 2nd arg = `{source_id: Population}` for cross-source joins. |
| `rule_spec` + no-code rule builder | authoring | Declarative rule (14 operators incl. a cross-source **exists_in/not_exists_in** primitive, AND/OR logic, severity, description template, item-key) → violations; cross-source rules render to plain-Python `test_code` so the bundle shape is unchanged. Same execution contract as Python. |
| Bundle + `contract/bundle.schema.json` | integration contract | The import shape the Uticen app consumes 1:1 (definitions + run provenance; no raw population). |
| Workpaper renderer (HTML + Markdown) | output | Self-contained, audit-grade workpaper: Results, Objective & Scope, Control, Data Sources, Procedures, Exceptions, Conclusion. A forked control renders **N procedures**, each headed **`{code} · {name}`** (e.g. "P1 · Segregation of Duties") with an **assertion subtitle**, a **per-check breakdown** listing each Test node's individual result, and a **union exceptions table** deduped by item-key with a **"failed check(s)"** column that names every check that flagged each item. Each procedure carries its own **per-procedure verdict pill** (evaluated against its own threshold), and the **overall control verdict** is an "any procedure fails ⇒ fail" roll-up. Multi-procedure section headings use the **dot separator** format "P1 · title" (earlier builds used a colon: "P1: title"). |
| `examples/northwind-trading/` | demo / template / fixture | A 9-control engagement (financial close / IT access / procurement / facilities, NIST 800-53 mapped) authored **entirely no-code** — two no-code rules + seven visual pipelines, several dropping to a **single Custom Python node** for an irreducible step (row-pairwise window, cross-source column arithmetic, column-to-column join compare) rather than a standalone Python script — to demonstrate the authoring ladder. One control (`datacenter-temperature`, PE-14) is built on a source **snapshotted from the public Open-Meteo API**, showcasing the "fetch from URL" on-ramp. Runnable demo, cold-user template, and end-to-end CI fixture. One control (`manual-je-review`, Finance.GL.1) is a **multi-procedure** showcase — it forks into **P1 · Segregation of Duties** (Custom Python) and **P2 · Reviewer Assigned** (no-code) over a shared materiality-filtered trunk, rolling two procedures into one control result. |
